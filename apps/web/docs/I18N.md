# i18n Guide — SkinSystem Dashboard

## Stack
- **Library**: `next-intl@4.9.1`
- **Source of truth**: `src/messages/en.json` (TypeScript types are inferred from this file)
- **Locales**: `pt` (default) · `es` · `en`
- **Locale resolution**: `proxy.ts` → header `x-locale` → `src/i18n/request.ts`
- **DEFAULT_LOCALE config**: `src/i18n/config.ts` — single source of truth. Import from here; never hardcode `'pt'` as a fallback string.

```ts
// src/i18n/config.ts
export const DEFAULT_LOCALE = 'pt' as const;
export const SUPPORTED_LOCALES = ['pt', 'es', 'en'] as const;
export type SupportedLocale = typeof SUPPORTED_LOCALES[number];
```

## Locale resolution chain

```
DASHBOARD_LOCALE cookie  (staff manual preference, written by setDashboardLocaleAction)
  → NEXT_LOCALE cookie   (browser manual choice from public language selector)
  → Accept-Language      (browser preference)
  → DEFAULT_LOCALE = 'pt'
```

**Never hardcode `?? 'es'` or `?? 'pt'` in page/layout files.**
Use `hdrs.get('x-locale') ?? 'pt'` only when passing locale as a prop to legacy APIs
that are still awaiting migration.

## Namespace structure

```
src/messages/
  ├── en.json   ← TypeScript source of truth (IntlMessages = typeof en)
  ├── pt.json   ← source text (real user language)
  └── es.json   ← translation

Namespace tree (dot-separated):
  dashboard.billing.*           DONE  (Phase 34, pilot)
  dashboard.customers.*         DONE  (Phase 35)
  dashboard.catalog.*           DONE  (Phase 35)
  dashboard.home.*              DONE  (Phase 35)
  dashboard.integrations.*      DONE  (Phase 35)
  dashboard.calendar.*          DONE  (Phase 35)
  dashboard.settings.*          DONE  (Phase 36)  ← sidebar, preview, brand, general, profile, team, preferences
```

Sub-namespace convention: `dashboard.<module>.<subSection>.<key>`
- Maximum 3 levels after `dashboard.`
- One sub-namespace per component or logical section — do not flatten everything
  into the module root.
- Use `camelCase` for all keys.

## Server Components (async)

```tsx
import { getTranslations } from 'next-intl/server';
import { headers } from 'next/headers';

async function MyPage() {
  const hdrs   = await headers();
  const locale = hdrs.get('x-locale') ?? 'pt';
  const t      = await getTranslations({ locale, namespace: 'dashboard.myModule' });

  return <h1>{t('title')}</h1>;
}
```

## Client Components

```tsx
'use client';
import { useTranslations, useLocale } from 'next-intl';

export function MyComponent() {
  const t      = useTranslations('dashboard.myModule.mySection');
  const locale = useLocale(); // use for date/currency/resolveI18n()

  return <button>{t('saveBtn')}</button>;
}
```

`NextIntlClientProvider` is mounted in `(dashboard)/layout.tsx` — all client
components in the dashboard tree inherit it automatically.

## Locale for date/currency formatting

```tsx
const INTL_LOCALE_MAP: Record<string, string> = {
  pt: 'pt-PT',
  es: 'es-ES',
  en: 'en-GB',
};
const intlLocale = INTL_LOCALE_MAP[locale] ?? 'pt-PT';

// currency
new Intl.NumberFormat(intlLocale, { style: 'currency', currency: 'EUR' }).format(cents / 100);

// date
new Date(iso).toLocaleDateString(intlLocale, { day: '2-digit', month: '2-digit', year: 'numeric' });
```

`locale` (from `useLocale()`) is the routing locale token (`'pt'`), **not** the
Intl BCP-47 tag (`'pt-PT'`). Always map through `INTL_LOCALE_MAP`.

## Dynamic DB content (nameI18n, descriptionI18n)

Service names, category names, and medical notes are stored as JSONB
`{ es: "...", en: "...", pt: "..." }` in the database.

**Do NOT put DB content in messages.json.** Use the `resolveI18n` helper:

```tsx
function resolveI18n(obj: unknown, locale: string): string {
  if (!obj || typeof obj !== 'object') return '—';
  const o = obj as Record<string, string>;
  return o[locale] ?? o['es'] ?? o['en'] ?? Object.values(o)[0] ?? '—';
}
```

## TypeScript types (zero config required)

`src/global.d.ts` declares `IntlMessages = typeof en`.
`useTranslations('dashboard.billing.history')` is fully type-checked — the
compiler will catch missing keys and typos automatically.

Never define a manual `interface BillingMessages`, `CatalogMessages`, etc.

## Key parity check (required after every migration)

Run from `apps/web/`:

```bash
node -e "
const pt = require('./src/messages/pt.json');
const es = require('./src/messages/es.json');
const en = require('./src/messages/en.json');
function flat(o, p='') {
  return Object.entries(o).flatMap(([k,v]) =>
    typeof v==='object' ? flat(v, p?p+'.'+k:k) : [p?p+'.'+k:k]);
}
const ns = 'dashboard.myModule';
const missingEs = flat(pt).filter(k=>k.startsWith(ns)&&!flat(es).includes(k));
const missingEn = flat(pt).filter(k=>k.startsWith(ns)&&!flat(en).includes(k));
console.log('missing es:', missingEs.length||'NONE');
console.log('missing en:', missingEn.length||'NONE');
"
```

## Anti-patterns to avoid

| Anti-pattern | Replace with |
|---|---|
| `_i18n.ts` local translation map | Delete it; move keys to `messages/*.json` |
| `const t = { es:{}, pt:{}, en:{} }[locale]` inline object | `getTranslations()` / `useTranslations()` |
| `locale` prop drilling into client components | `useLocale()` in the client component |
| `useCatalogT()` via TenantProvider | `useTranslations()` |
| `import msgs from '@/messages/es.json'` + manual indexing | `useTranslations()` |
| `stripeT(locale)` custom bridge file | `getTranslations({ locale, namespace: 'integrations.stripe' })` |
| `?? 'es'` or `?? 'pt'` locale fallback | `?? DEFAULT_LOCALE` imported from `@/i18n/config` |
| `STATUS_LABEL: Record<string, Record<string, string>>` inline | Extract to `messages/*.json`, consume via `t('statusPending')` |
